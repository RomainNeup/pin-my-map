# TAS-35 + TAS-36 ‚Äî Register/Login with Apple and Google

**Epic:** User  ‚Ä¢  **Priority:** Bonus  ‚Ä¢  **Status:** back-ready
**Branch:** `tas-35-36-oauth`

## Goal

Users can sign up / sign in via Apple ID and Google. Both providers issue an OpenID Connect ID token; we verify and use it to find or create a local `User`.

## Backend (single agent: back-oauth)

### Schema

`User` gains:
- `@Prop({ required: false, index: { sparse: true } }) googleId?: string`
- `@Prop({ required: false, index: { sparse: true } }) appleSub?: string`
- `password` becomes optional (`required: false`) ‚Äî OAuth-only users have none. AuthService.register existing flow still requires password for the email path.

### Routes

| Method | Path | Auth | Notes |
|--------|------|------|-------|
| POST | /auth/oauth/google | public, throttled 10/min | body `{ idToken: string }`, 200 LoginResponseDto |
| POST | /auth/oauth/apple | public, throttled 10/min | body `{ idToken: string, name?: string }` (Apple only sends name on first sign-in), 200 LoginResponseDto |

Both: verify the ID token via the provider's JWKS (use `jose` ‚Äî already a dependency). Match audience to env vars `GOOGLE_OAUTH_CLIENT_ID` / `APPLE_OAUTH_CLIENT_ID`. On verified token: find by provider sub; else find by email and link; else create new user (status follows `registrationMode`: open ‚Üí active, approval-required ‚Üí pending, invite-only ‚Üí 403 unless email is on an existing user). Set `name` from the token payload (Google: `name`; Apple: from request body or fallback to email local-part).

### DTOs

```ts
export class GoogleOAuthDto { @IsString() @IsNotEmpty() idToken: string; }
export class AppleOAuthDto { @IsString() @IsNotEmpty() idToken: string; @IsOptional() @IsString() name?: string; }
```

### Service

`AuthService.loginWithGoogle(idToken)` and `AuthService.loginWithApple(idToken, name?)` ‚Äî share a private `findOrLinkOrCreate(provider, sub, email, name, status)` helper.

### Audit

`auth.oauth.login` and `auth.oauth.register` (when a new user is created).

### Env

Add to `back/.env.example`:
- `GOOGLE_OAUTH_CLIENT_ID=`
- `APPLE_OAUTH_CLIENT_ID=`

Verification fails if the env var is missing ‚Äî return 503 with a clear message ("OAuth provider not configured").

### Tests

Unit tests using mocked JWKS verifier: happy path, invalid token, audience mismatch, find existing by sub, link existing by email, create new (each registrationMode), invite-only blocks new.

## Frontend (single agent: front-oauth)

### UI

- `/login` and `/register`: "Continue with Google" and "Continue with Apple" buttons under the email form, separated by a divider. Use brand-correct colors and icons (small inline SVGs are fine).
- Google: use `@react-oauth/google` is React-only ‚Äî instead use the official Google Identity Services script (`https://accounts.google.com/gsi/client`) loaded once. On callback, POST `idToken` to `/auth/oauth/google`, then store JWT and redirect to `/`.
- Apple: use Apple JS (`https://appleid.cdn-apple.com/appleauth/static/jsapi/appleid/1/en_US/appleid.auth.js`). On success, POST `id_token` (and `user.name` if present on first call) to `/auth/oauth/apple`.

### Env

Add to `front/.env.example`:
- `PUBLIC_GOOGLE_OAUTH_CLIENT_ID=`
- `PUBLIC_APPLE_OAUTH_CLIENT_ID=`
- `PUBLIC_APPLE_OAUTH_REDIRECT_URI=`

When env vars are missing, the OAuth buttons are hidden (graceful degradation in dev).

### Files

- `front/src/lib/oauth/google.ts` (NEW) ‚Äî script loader + sign-in trigger
- `front/src/lib/oauth/apple.ts` (NEW) ‚Äî script loader + sign-in trigger
- `front/src/lib/api/auth.ts` ‚Äî add `oauthGoogle(idToken)`, `oauthApple(idToken, name?)`
- `front/src/routes/(auth)/login/+page.svelte` and `register/+page.svelte` ‚Äî buttons
